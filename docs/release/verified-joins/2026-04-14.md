## CC side (rehearsed on ubuntu dev box)

### 1. Install

```
$ bun run build:cli && bun run build:plugin
$ npm pack                         # produced firstintent-a2a-bridge-0.1.0.tgz
$ npm i -g ./firstintent-a2a-bridge-0.1.0.tgz
added 3 packages in 538ms

$ a2a-bridge --version
a2a-bridge v0.1.0
```

### 2. Init

`a2a-bridge init` without flags refuses because `codex` is not on the
PATH (the default CC install checks Codex as a peer dep).  `init --print`
skips the dep gate:

```
$ a2a-bridge init --print
Wrote /home/ubuntu/.local/state/a2a-bridge/config.json
Bearer token (32 bytes, hex): <redacted>
…
# OpenClaw (acpx.config.agents)
{
  "agents": {
    "a2a-bridge": {
      "command": "a2a-bridge",
      "args": ["acp"]
    }
  }
}
```

### 3. Daemon start

`a2a-bridge daemon start` blocks on `/readyz` which in v0.1 returns
`503` until Codex bootstraps — so on a Codex-less dev box the
managed-start times out.  Running the daemon directly
(`bun run src/runtime-daemon/daemon.ts`) brings the control plane
up anyway:

```
$ bun run src/runtime-daemon/daemon.ts >/tmp/a2a-daemon.log &
$ curl -s http://127.0.0.1:4512/healthz
{"bridgeReady":false,"tuiConnected":false,"threadId":null,
 "queuedMessageCount":0,"proxyUrl":"ws://127.0.0.1:4511",
 "appServerUrl":"ws://127.0.0.1:4510","pid":1081744}
```

Control plane listens on `ws://127.0.0.1:4512/ws` — this is what
`a2a-bridge acp` on the ACP-client side connects to.
