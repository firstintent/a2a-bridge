`a2a-bridge init` without flags refuses because `codex` is not on the

skips the dep gate:

### 3. Daemon start

`a2a-bridge daemon start` blocks on `/readyz` which in v0.1 returns
`503` until Codex bootstraps — so on a Codex-less dev box the
managed-start times out.  Running the daemon directly
(`bun run src/runtime-daemon/daemon.ts`) brings the control plane
up anyway:

```
$ bun run src/runtime-daemon/daemon.ts >/tmp/a2a-daemon.log &
$ curl -s http://127.0.0.1:4512/healthz
{"bridgeReady":false,"tuiConnected":false,"threadId":null,
 "queuedMessageCount":0,"proxyUrl":"ws://127.0.0.1:4511",
 "appServerUrl":"ws://127.0.0.1:4510","pid":1081744}
```

Control plane listens on `ws://127.0.0.1:4512/ws` — this is what
`a2a-bridge acp` on the ACP-client side connects to.

### 4. What this box cannot do

continues the verification on a laptop where a real Claude Code

## Notes / fixes fed back

- The Codex-missing gap on `init` / `daemon start` is a pre-existing
  v0.1 limitation (the default config expects Codex as a peer); it is
  **not** an ACP-side issue and does not affect the CC→OpenClaw
  round-trip when both peers are correctly provisioned.  Follow-up:
  expose a `--no-codex` flag so bridge-only installs don't need a
  Codex binary.
- Cross-host support required two new env vars committed in `cbeac78`:
  `A2A_BRIDGE_CONTROL_HOST` (daemon bind) and
  `A2A_BRIDGE_CONTROL_URL` (ACP subprocess target). Both default to
  localhost so existing same-host behavior is unchanged.
- First pass used a stub auto-responder; second pass used a **real
  Claude Code** session (v2.1.105, Sonnet 4.6) with the a2a-bridge
  channel plugin loaded via `a2a-bridge claude` in a tmux session.
  The real CC replied with genuine reasoning, proving the full
  production wire path end-to-end.
- The `/readyz` endpoint previously gated on Codex bootstrap,
  blocking the plugin from connecting on Codex-less machines. Fixed
  in this session by always returning 200 once the control plane is
  listening (committed alongside this transcript).
- The `reply` tool required manual approval in CC on the first call;
  selecting "Yes, and don't ask again" resolved this for subsequent
  turns. First-time users of `a2a-bridge claude` will hit this once.
